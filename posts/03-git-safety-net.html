How to Recover a Deleted Git Commit or Branch with Reflog

Published on June 30, 2026 • Updated on July 23, 2026 • By Tweaknook Editorial

A deleted local branch or an accidental git reset --hard does not always mean the commit is gone. Git usually records recent movements of HEAD in the local reflog, which can help you find the previous commit and create a rescue branch.

Important: Reflog is local to your repository. It cannot recover an untracked file that was never committed or stashed, and old entries may eventually expire.

Step 1: Check the Recent Reference History

Open a terminal inside the affected repository and run:

git reflog

A simplified result may look like this:

a1b2c3d HEAD@{0}: reset: moving to HEAD~1
e5f6a7b HEAD@{1}: commit: Add checkout layout
9a8b7c6 HEAD@{2}: checkout: moving from main to dev

The values at the left are abbreviated commit hashes. In this example, e5f6a7b is the commit immediately before the reset.

Step 2: Inspect the Commit Before Restoring It

Confirm that the commit contains the work you expect:

git show e5f6a7b

Review the author, date, message, and changed files before continuing.

Step 3: Create a Rescue Branch

Create a new branch at the recovered commit instead of moving your current branch immediately:

git switch -c rescue-branch e5f6a7b

On older Git versions, use:

git checkout -b rescue-branch e5f6a7b

Your recovered work is now attached to rescue-branch. You can test it and then merge or cherry-pick the commit into the correct branch.

Recover a Deleted Local Branch

If a branch was deleted but its commit still appears in reflog, recreate it with:

git branch recovered-work e5f6a7b

This creates the branch without changing your current working branch.

What Reflog Can and Cannot Recover

SituationTypical result
Committed work removed by a resetOften recoverable while the reflog entry and Git object still exist
Deleted local branchOften recoverable if its commit can be found
Untracked file never committed or stashedNot recoverable through Git reflog
Repository copied to another computerThe original computer's reflog is not transferred by a normal clone

Safer Habits for Future Work

  • Create a temporary branch before destructive resets.
  • Use git status before deleting or resetting.
  • Commit or stash important work before major changes.
  • Push important branches to a remote repository as an additional copy.

Reflog is a useful recovery tool, but it is not a substitute for commits and remote backups. The safest approach is to inspect the reflog, verify the target commit, and recover it into a new branch.

If the Commit Does Not Appear

First confirm that the terminal is open in the correct repository. Then check references from all local branches with:

git reflog --all

A reflog entry may be missing if the work was never committed or stashed, the recovery is being attempted on a different computer, or the entry has expired. Avoid running cleanup commands while you are still investigating because unreachable Git objects can eventually be removed.
